# 2023-12-15
- Today's programming wisdom is to always prioritize code readability and maintainability. This means writing concise, well-structured, and easy-to-understand code that can be easily modified and expanded upon as needed. To achieve this, follow these best practices:

1. Use clear variable, function, method, class, module, etc. names that accurately represent their purpose and functionality.
2. Follow coding conventions and standards for the language and framework you're using. These guidelines exist to ensure consistency and reduce the risk of errors. They may include naming conventions, indentation, comment style, code structure, et cetera. Check the official documentation for your preferred language or framework for specific guidelines. 
3. Break up your code into logical, reusable, self-contained units called functions, methods, classes, modules, or whatever your language/framework calls them. Doing so makes your program easier to understand, maintain, test, debug, reuse, scale, document, teach, eat, sleep, drink, breathe, you get the idea. Functions should be small, single-purpose, easy to reason about, have well defined inputs and outputs, use the least amount of global state as possible, return values when appropriate, throw exceptions when an error occurs, handle errors gracefully, log errors, validate input, perform side effects, do anything else your domain requires, but should never do more than necessary. Methods should follow the same principles as functions but are usually a part of a class and are therefore more closely tied to the class' behavior. Classes should have a single responsibility and follow SOLID principles. Modules should provide a cohesive set of related functionality and be self contained. Etc., get it?
4. Comment your Code. Write comments that explain what the code does, why it' s there, how it works, what it does not do, side-effects, anything that might not be obvious to someone reading your (future) code. Comments should not just be a recap of the variable names or function names. Good comments are concise and to-the-point. Bad comments go on for paragraphs and make the reader want to scream. Don't be afraid to remove comments if they' re no longer needed or if you can explain the logic more concisely. Remember, comments should help the person reading the source code, not replace the need for clear code in the first place. Also, don' t be tempted to comment out code just because it doesn' thave comments yet. That' ll only make it harder to add comments later. Instead, put it off until you have more time to write a good comment. Or ask for help. There are people out there who want you to succeed. And they may even be able to explain your logic better than you. So don t hesitate to ask. It'll make you a better programmer. Promise.

First, prioritize your tasks. When you're working on a project, it' s important to determine which tasks are most critical to the success of the project. By focusing on these tasks first, you can ensure that your code is of high quality and meets the requirements of your project goals. You can use the MoSCoW method to help prioritize: Must have, Should have , Could have and Won't have. Must-have tasks should be completed first to ensure the basic functionality of a product or project is met. Should-haves are nice-to- have features that add value to a solution but don' t impact the core functionality. Could-has and won'ts are tasks that can be done later or not done at all without harming the overall project success. 
Next, planning ahead is crucial. Before starting a new project or a task, take some time to plan out your approach. Write down the steps you plan to take and the resources you will need. Then, map out a timeline for each step and set deadlines for yourself. Planning ahead will ensure you have a clear understanding of what needs to get done and when it needs t o be complete. It will also help prevent last-minute panic and stress.
